## 二、高频面试题 — 实战对话版

### 1️⃣ "说说你最拿手的一个项目"

**不要这样答**："我做过一个低代码引擎，支持拖拽…有 AST 代码生成…" —— 这是在念简历。

**这样答**（3 段式）：

> "我选低代码表单引擎这个项目来说吧。它的核心要解决一个问题：中后台系统里表单开发占据大量重复劳动，产品一天改三版，开发跟着改 Schema 改到崩溃。
>
> 我的方案是做一个**配置驱动的表单渲染引擎**。设计器端用拖拽搭建 JSON 协议，消费端拿到 JSON 直接渲染。技术上有三个难点：
>
> **第一个是递归渲染**。表单不是扁平的——分组里可以嵌套栅格，栅格里再嵌套分组。如果写死两层循环，产品要求加第三层就炸了。我用了自顶向下递归的设计，`FormNode` 组件自己调用自己，理论上无限层级，实际用 maxDepth 做安全兜底。代码在 `apps/vue-app/src/components/FormNode.vue`。
>
> **第二个是 AST 代码生成**。产品说"我不要只看预览，我要拿到能跑的 Vue 代码"。最粗暴的做法是字符串拼接——但很容易拼错标签、少引号，而且没法做语法校验。我借鉴了编译原理 AST 的思想：遍历 component 树，按类型分派生成器，最后组合成三段式 SFC。代码在 `apps/vue-app/src/utils/simpleAstGenerator.ts`，我还给核心模块补了 62 个 Vitest 单元测试。
>
> **第三个是校验体系**。低代码不能只管生不管养——用户填表单时必须校验。我做了 11 种验证规则的动态生成器，从 JSON Schema 自动映射到 Element Plus 的 el-form rules，代码在 `apps/vue-app/src/utils/validation.ts`。
>
> 这个项目给我最大的收获不是技术本身，而是**协议设计的能力**——一个好的 JSON Schema 结构可以让渲染端和设计端完全解耦，甚至未来换框架都不需要改协议。"

### 2️⃣ "为什么从 Vue 3 切到 React 19？"

**核心纠正**：你的项目不是"迁移"，是"双框架共存"。这个说法面试官更喜欢。

> "严格来说不是迁移，是**架构升级**。我们原来的低代码引擎是纯 Vue 3，后来来了一个客户需要用 React 版的管理后台。如果重写一套，两个版本的功能永远不同步。
>
> 我的解法是：把核心算法抽到 `packages/core`，包括类型定义、验证规则生成、可见性表达式求值、资产管理业务逻辑。这个 core 包**完全不依赖任何 UI 框架**，纯 TypeScript 纯函数。然后 Vue 和 React 两端各自用适配层去消费它。
>
> 踩过最大的坑：**响应式机制不兼容**。Vue 里 ref 是 Proxy 代理对象，React 里 useState 是闭包快照。在 core 包里不能直接用 reactive，必须返回纯数据，由各端自己包装。另一个坑是拖拽——Vue 用 vuedraggable，React 用 @dnd-kit，API 完全不同，但拖拽结束后的数据结构操作完全一样，这部分就天然适合放 core 里。"

### 3️⃣ "你手写的 VirtualTable 和 react-window 有什么区别？"

> "react-window 是个很好的通用库，但我们的场景有三个特殊需求它不满足：
>
> **第一，多层嵌套表头**。资产管理系统里设备分组有三级分类，表头需要做合并单元格。react-window 只支持平铺表头。
>
> **第二，动态行高**。有些行有备注字段需要展开，行高不固定。我用了一个行高缓存 Map，先按预估高度 + 20px 余量渲染，渲染完成后用 ResizeObserver 回调修正实际高度，再触发一次微调。
>
> **第三，React 19 并发模式的适配**。这是踩坑最深的点。虚拟列表依赖精确的 scrollTop 和可视窗口计算，但 React 19 的 concurrent rendering 可能中断渲染，导致 startIndex 和实际可见行不对齐，出现闪烁。解决方式是用 `useRef` 维护滚动状态而非 `useState`，因为 ref 的更新是同步的不受并发中断影响。
>
> 最终效果：10000 条数据渲染从 3.2 秒降到 16ms 首帧，滚动帧率稳定 60fps。"

### 4️⃣ "Token 无感刷新和并发请求控制怎么做的？"

> "我们的接口层在 `packages/shared/src/api/request.ts`，基于 Axios 做了深度封装。
>
> **Token 管理**：登录成功后 JWT 存在 localStorage 里，请求拦截器自动挂到 Authorization 头。当服务器返回 401 时，响应拦截器清除 token 并跳转登录页。
>
> 目前是简易版——真正的无感刷新需要一个 `isRefreshing` 标志位：当第一个请求收到 401 时，暂停它和所有后续请求，用 refreshToken 换新 token，成功后重放所有暂停的请求。我的代码里目前是直接踢掉，这是一个已知的待优化点。
>
> **并发竞态**：这个反而在我的 React 版资产管理系统里实现了。用户狂点提交按钮会导致产生多个相同的 POST 请求，后端收到一堆脏数据。我用 `AbortController` + `Map<requestKey, controller>` 实现：相同 key 的请求如果上一个还没完成，直接 `abort()` 掉旧的，只保留最新的。
>
> 有一个细节：不是所有请求都应该取消——GET 请求 cancel 了只是不发，但 DELETE 请求 cancel 了可能数据已删。所以在 Map 里只存 mutate 类型（POST/PUT/DELETE）的请求。"

### 5️⃣ "你在项目中遇到过最难排查的 Bug 是什么？"

> "选一个真实的：**低代码设计器里 select 组件的 options 更新后下拉列表不刷新**。
>
> 现象：用户给下拉选择器加了新选项，右侧属性面板已经显示新数据，但点击下拉还是旧的选项。
>
> 排查过程：先用 Vue DevTools 看组件 props，发现 options 已经更新了。然后在 el-select 的源码里打断点，发现组件内部有个 `cachedOptions` 没有触发 invalidate。
>
> 根因：Element Plus 的 el-select 对 options 做了浅比较，而我们用了 `options.push(newItem)`——数组引用没变，组件认为没更新。
>
> 解决方案：改为 `options = [...options, newItem]`，每次都创建新数组引用。
>
> 这个 Bug 教会我一件事：**框架的响应式机制不是你写了它就会更新，你必须理解它内部怎么判断"变了"**。Vue 对 ref 的深层追踪有边界，数组的变异方法在某些场景下不会触发更新——这和 Vue 2 的 `$set` 问题本质一样，只是表现形式不同。"

### 6️⃣ "你的低代码引擎的 AST 生成器和真正的编译器有什么区别？"

> "诚实说，我的 SimpleASTGenerator 不是真正的编译器，它是一个**借鉴了 AST 思想的代码生成器**。
>
> 真正的编译流程是：源码 → 词法分析 → 语法分析 → AST → 代码生成。我的流程是：JSON Schema → 遍历 component 树 → 分派生成器 → 拼接字符串。
>
> 但它借鉴了 AST 的几个关键思想：一是**结构化的中间表示**——处理嵌套组件时不是暴力拼接，而是递归构建模板、脚本、样式三个部分再组合；二是**上下文传递**——生成子组件时需要知道自己在第几层缩进，这个上下文通过参数传递而非全局变量。
>
> 我们正在做真正的 AST 方案升级——用 @babel/parser 生成标准 AST，好处是可以利用 Babel 生态做代码优化、静态分析、甚至转译到不同版本的 Vue。方向在 `AST_GENERATOR_UPGRADE.md` 里有详细规划。
>
> 面试官如果追问"为什么不一开始就用 AST"——我会说：先跑通核心流程验证价值，再引入编译技术提升质量，这是工程上的务实选择。"

### 7️⃣ "你对 TypeScript 的掌握到什么程度？"

> "我在项目中用的都是业务里真正需要的 TS 特性，不炫技：
>
> 一是**泛型约束**。`filterAssets<T extends AssetProject>` 保证传入的类型必须有 id、category、status 等字段，同时允许子类型扩展。
>
> 二是**联合类型和类型守卫**。`AssetStatus = 'active' | 'archived' | 'repair' | 'pending' | 'scrapped'`，状态机只能在这些值之间跳转，类型系统帮我在编译期就拦住非法状态。
>
> 三是**工具类型**。`Partial<BaseNode>` 用于节点更新，`Pick` 和 `Omit` 用于 API 请求体的类型控制。
>
> 踩坑点：递归类型推断。FormSchema 的 components 字段是 `FormComponent[]`，而 FormComponent 里又有 `list?: FormComponent[]`——TS 在推导这种自引用类型时可能卡死。我的解法是用 `interface` 而非 `type` 定义递归结构，TS 对 interface 的惰性求值处理更好。"

---

### 8️⃣ "Webpack 和 Vite 的区别？"

> "Webpack 先打包再启动，Vite 基于 ESM 按需编译，所以冷启动快。
>
> 但实际开发中的感受更具体：
>
> 我们项目里 HMR 的体验差别很大。改一个 Vue 组件的 color，Vite 能做到 20ms 内热更新，Webpack 要 2-3 秒——因为 Vite 只 invalidate 改动的那个模块链，Webpack 要重新编译整个 chunk。
>
> 不过 Vite 也有坑。我们今天调试时遇到一个 'App.css renamed to App.css' 的警告，这是 Vite 在 Windows 路径处理上的 bug——大小写不敏感的文件系统导致同一个文件被当成两个模块注册。这个 Bug 在生产环境不影响，但开发时很烦。
>
> 另外 Vite 的预构建阶段用了 esbuild，esbuild 的 TS 编译不做类型检查——所以开发阶段 TS 类型错误不会报，直到 `vite build` 才暴露。我的解决方式是在 tsconfig 里开 `noEmit + strict`，开发时 IDE 提示类型错误就及时修。"

## 三、踩坑经验集（面试杀手锏）

这些是你可以主动抛出的真实经历，面试官会觉得你很实在：

### 坑 1：pnpm workspace 幽灵依赖
> "配置了 `packages/core` 依赖 `typescript` 作为 devDeps，但没在 core 的 package.json 里声明，而是在根 package.json 里声明的。pnpm 的严格模式不允许跨包访问未声明的依赖——会直接报 module not found。解决方法是每个子包单独声明自己的依赖，依赖提升只靠 workspace protocol。"

### 坑 2：Element Plus TreeShaking 失效
> "用了 `unplugin-vue-components` 做按需引入，但构建产物里 el-button 和 el-input 还是被打包了。排查发现是因为在 `FormRenderer` 里用了动态组件 `<component :is="xxx">`，静态分析追踪不到实际引用。解决方式是在 `componentMap` 里显式 import 所有需要的组件。"

### 坑 3：React StrictMode 双重渲染
> "开发模式下 React.StrictMode 会故意渲染两次来检测副作用。虚拟列表的 scrollTop 计算在两次渲染间被重置，导致位置抖动。发现后在 `useEffect` 的清理函数里做了状态恢复。这个坑提醒我 StrictMode 不是摆设——如果一个组件在 StrictMode 下有问题，生产环境迟早也会出。"

### 坑 4：@dnd-kit 拖拽上下文丢失
> "今天刚修的一个 Bug。MaterialPanel 里的 `useDraggable` 和 CanvasPanel 里的 `useDroppable` 需要同一个 `DndContext`。我一开始把 Context 放在 CanvasPanel 里，MaterialPanel 在 Context 外面，拖拽不生效。React 的 Context 机制不跨组件树分支——只在 Provider 子树内有效。提升到共同父组件 `Designer` 后解决。"

### 坑 5：Antd 6.x 破坏性变更
> "项目升级到 antd 6 后，所有 `Space direction='vertical'` 都报 warning。antd 6 把 `direction` 改名为 `orientation`。这种 API 重命名的坑用 ESLint 规则 `@typescript-eslint/no-deprecated` + antd 的 codemod 可以批量修，但小型项目手动改更可控。"

## 四、面试常见陷阱问题

### 💀 "你的项目有什么缺点？"

**不要说**："我觉得做的挺好的"

**这样说**：
> "有两个。第一是测试覆盖率不够——我给 packages/core 补了 62 个单元测试，但 vue-app 和 react-app 里的 UI 组件还没有 e2e 测试。第二是错误处理不够优雅——很多地方是 console.error + ElMessage.error，缺少统一的错误边界和 Sentry 集成。这些都是我接下来要补的。"

### 💀 "你和后端意见不一致怎么办？"

> "举一个真实例子。后端同事设计接口时把表单 Schema 存成字符串 JSON，每次取用都要 `JSON.parse`，而且数据库里无法查询 Schema 内部字段。我建议他用 JSONB 类型（PostgreSQL）或者至少存一个 version 字段方便做版本管理。他解释说是为了兼容 MySQL。最后折中方案是：存 JSON 字符串但加一个 `schema_version` 和 `checksum` 字段，前端发 Schema 时带上版本号，后端做乐观锁校验。这样双方都有保障。"

### 💀 "你的期望薪资是多少？"

> "我先了解下咱们这边的薪资结构和晋升机制吧，因为我更看重成长空间。如果合适的话，薪资可以谈。"
