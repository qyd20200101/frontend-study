## 一、四个阶段完整链路（原🔴：卡在用户打字→DOM 更新）

```
模板字符串 .vue
    │  ① 编译（compile，构建时 vue-loader / vite-plugin-vue 完成）
    ▼
render 函数（h(...) 嵌套调用）+ 静态提升 + patchFlag + cache
    │  ② 挂载（mount，首次执行 render → 拿到 vnode → patch(null, vnode)）
    ▼
真实 DOM + effect 关联（render 函数本身就是一个 effect）
    │  ③ 渲染（render，每次 effect 重跑，产出新 vnode）
    │  ④ 更新（patch，新旧 vnode diff → 局部 DOM 操作）
    ▲
    └── 数据变化 → trigger → queueJob → 微任务里跑 effect → 回到 ③
```

**一句话**：编译是构建时的事（产出 render + 标记），后三个是运行时循环。

## 二、Vue2 vs Vue3 响应式（原🟡）

| 维度 | Vue2 | Vue3 |
|---|---|---|
| API | `Object.defineProperty` 劫持每个 key | `Proxy` 代理整个对象 |
| 新增属性 | 检测不到 → `Vue.set` / `this.$set` | 自动检测 |
| 删除属性 | 检测不到 → `Vue.delete` | 自动检测 |
| 数组 | 重写 7 个变更方法（push/pop/shift/unshift/splice/sort/reverse） | Proxy 直接拦截索引和 length |
| 初始化 | 递归遍历所有 key（深度） | 惰性，访问时才代理子对象 |
| Map/Set | 不支持 | 支持 |

**记忆口诀**：Vue2 是"门口装监控"（每个 key 装一个），Vue3 是"包整栋楼"（Proxy 一层包住）。

## 三、effect / dep / track / trigger（原🟡）

对照自己的 [Mini_ref.ts](../../../js/day30-js底层/Vue3从项目学源码/Mini_ref.ts)：

```
你的代码                  真实 Vue3 源码
──────────────────────────────────────
this.dep (Set<Fn>)       →   dep (Set<ReactiveEffect>)
activeEffect            →   activeEffect (全局变量)
get value() {dep.add}   →   track(target, key)          ← 收集
set value() {dep.fEach} →   trigger(target, key)        ← 派发
effect(fn)              →   new ReactiveEffect(fn).run()
```

**链路**：
```
读 .value → getter → track → dep.add(activeEffect)
写 .value → setter → trigger → dep.forEach(eff => scheduler(eff))
```

**reactive 多一层**：`Map<target, Map<key, Set<effect>>>` 三层嵌套（叫 targetMap），因为对象有多个 key。ref 只有一个值，所以一个 dep 就够了。

## 四、编译期 3 大优化（原🟢，再钉一遍，防止第四次搞混）

| 优化 | 干什么 | 例子 |
|---|---|---|
| **静态提升 hoistStatic** | 把不会变的 vnode 抽到 render 外面，复用同一引用 | `<p>foo</p>` → 提到模块顶层 `_hoisted_1` |
| **patchFlag** | 给动态节点打位标记，diff 时按位跳过静态部分 | `<p>{{x}}</p>` → `1 /* TEXT */`；`<p :id="x">` → `8 /* PROPS */` |
| **事件缓存 cacheHandlers** | inline 事件包一层缓存，避免每次 render 创建新函数 | `@click="x++"` → `_cache[0] \|\| (_cache[0] = $event => x++)` |

**顽固盲区提醒**：这三个都在**编译期**做，diff 是**运行时**的消费者。编译管创建，diff 管比对。

## 五、diff 双端 + LIS（原🟢）

```
patchKeyedChildren(c1, c2):
  ① 头头比对：i 从 0 开始，相同就 patch 并 i++
  ② 尾尾比对：e1/e2 从末尾开始，相同就 patch 并 e--
  ③ 头尾剩空：仅有挂载或卸载
  ④ 中间乱序段：
      - 建 keyToNewIndexMap（新中间段的 key → 索引）
      - 遍历旧中间段：找到的 patch + 记录新位置；找不到的 unmount
      - 对新位置数组求 LIS（最长递增子序列）
      - 倒序遍历新中间段：在 LIS 里的不动，不在的 move
```

**为什么用 LIS**：保证 DOM move 操作次数最少。

## 六、scheduler / nextTick（原🟡）

```js
trigger(...) → dep 里每个 effect 走 scheduler → queueJob(effect)

queue: []  (有去重，同 effect 不重复入队)
flushPromise = Promise.resolve()
flushPromise.then(flushJobs)   // 微任务
```

**nextTick(cb)** = `Promise.resolve().then(cb)`，和 scheduler 共享一个微任务队列，但 effect 已经先入队，所以 cb 拿到的是更新后 DOM。

```js
ref.value++; ref.value++; ref.value++;   // 同步 3 次
// effect 只入队一次 → 微任务里只跑一次 render
```

## 七、computed vs watch（原🟡 —— 原自评"带返回值的 effect"只对一半）

| | computed | watch |
|---|---|---|
| 本质 | 一个**带缓存的 effect** | 一个**手动控制的 effect** |
| 返回值 | 返回 ref（`.value` 访问） | 返回 stop 函数（用来停止监听） |
| 何时执行 | 读 `.value` 时按需算（lazy + dirty 标记） | 依赖变就立即（或 flush 时）跑回调 |
| scheduler | `dirty = true`，不立即重算 | 把 cb 推入 scheduler 队列 |

**原自评修正**：
- "computed 是带返回值的 effect" → 对了一半，关键在**缓存（dirty 标记）+ 返回 ref**
- "watch 没返回值" → 错，watch 返回 stop 函数；只是它没把"算出来的值"作为返回值

```ts
const sum = computed(() => a.value + b.value)  // sum 是 ref
const stop = watch(a, (n, o) => {...})         // stop 是函数
```

## 八、生命周期（原🔴："很混乱"）

**组件钩子顺序**（父子嵌套，口诀："父开门让子进来，子先穿衣，父后穿衣"）：

```
父 setup
父 onBeforeMount
  子 setup
  子 onBeforeMount
  子 onMounted        ← 子先挂载完
父 onMounted          ← 父后挂载完

更新时：
父 onBeforeUpdate
  子 onBeforeUpdate
  子 onUpdated
父 onUpdated

卸载时：
父 onBeforeUnmount
  子 onBeforeUnmount
  子 onUnmounted
父 onUnmounted
```

**指令钩子 7 个**：`created → beforeMount → mounted → beforeUpdate → updated → beforeUnmount → unmounted`

**关键易错（原自评记反了）**：
- ❌ 原自评写的："指令 mounted 晚于组件 onMounted 触发"
- ✅ 正确："指令 mounted 先于组件 onMounted 触发"

顺序：**指令 mounted → 组件 onMounted**（子先父后，指令算"更靠近 DOM 的子"）

## 九、v-指令的等价转换（原🔴）

```html
<!-- v-if -->
<div v-if="ok">A</div><div v-else>B</div>
↓ 编译为
ok ? h('div', 'A') : h('div', 'B')

<!-- v-for -->
<li v-for="(item, i) in list" :key="item.id">{{item.name}}</li>
↓ 编译为
h(Fragment, list.map((item, i) => h('li', { key: item.id }, item.name)))

<!-- v-model（Vue3 组件上） -->
<Child v-model="msg" />
↓ 等价于
<Child :modelValue="msg" @update:modelValue="v => msg = v" />

<!-- v-model（input 上） -->
<input v-model="msg" />
↓ 等价于
<input :value="msg" @input="e => msg = e.target.value" />
```

**Vue2 vs Vue3 的 v-model 区别**：
- Vue2：组件上是 `:value` + `@input`，改名要用 `model` 选项；只能一个
- Vue3：默认 `:modelValue` + `@update:modelValue`；支持多个 `v-model:title` `v-model:content`；改名直接 `v-model:foo`

## 十、v-for 不能 `:key="index"`（原🟡）

```html
<li v-for="(item, i) in list" :key="i">{{ item.name }}</li>
```

**问题**：list 头部插入新元素时，所有原元素的 index 后移，导致 Vue 以为每一项都"变了"，diff 走 patch 而不是 move，**输入框失焦、状态丢失、动画错乱**。

**正确**：用业务 id（`item.id`），让 diff 能识别"还是同一个节点，只是位置变了" → 走 move 而不是重建。

## 十一、TS 最小够用集（原🔴："一窍不通"）

```ts
// 1. 泛型：函数/类的"类型参数"
function id<T>(x: T): T { return x }
id<number>(1)   // T = number

// 2. 约束 extends：T 必须是某种形状
function getLen<T extends { length: number }>(x: T) { return x.length }
getLen('hi')        // ✓ string 有 length
getLen([1,2])       // ✓ array 有 length
getLen(123)         // ✗ number 没 length

// 3. 工具类型（背 5 个）
Partial<T>          // 全部可选
Required<T>         // 全部必填
Pick<T, K>          // 挑选 K 这几个 key
Omit<T, K>          // 排除 K 这几个 key
Record<K, V>        // 等价 { [k in K]: V }

// 4. "接口传参取固定值"场景：字面量联合 + 泛型约束
type Theme = 'light' | 'dark'
function setTheme(t: Theme) {...}    // 只能传这两个值

// 进阶：从 obj 里只允许传它的某个 key
function get<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key]
}
const u = { name: 'a', age: 1 }
get(u, 'name')   // ✓ 类型是 string
get(u, 'xxx')    // ✗ 编译报错

// 5. interface vs type
interface User { name: string }   // 可被 extends/implements，可声明合并
type User = { name: string }      // 可联合/交叉，不可合并
```

**关键**：Day12 学到的 `T extends U` 是"**约束**"，不是"继承" —— TS 里 `extends` 在泛型位置 99% 都是约束。
