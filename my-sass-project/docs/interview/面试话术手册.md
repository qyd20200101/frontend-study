## 一、浏览器八股

### 1. 从输入 URL 到页面渲染全过程

**思路**: DNS → TCP → TLS → HTTP → 解析 HTML → 构建 DOM/CSSOM → 布局 → 绘制 → 合成

**口语回答**:
"整个过程可以分网络和渲染两大块。网络层面，先 DNS 解析域名，拿到 IP 后走 TCP 三次握手，如果是 HTTPS 再加 TLS 握手。然后浏览器发送 HTTP 请求，服务器返回 HTML。

拿到 HTML 后进入渲染流程：HTML 解析器构建 DOM 树，遇到 CSS 资源时解析构建 CSSOM，这两个都是树结构。两棵树合并成 Render Tree，然后进入 Layout 计算每个节点的精确位置和大小，最后 Paint 绘制像素。

我的项目中，vite.config.ts 里配置了 Gzip 压缩和 chunk 拆分，目的就是减少网络传输体积，直接加速第一阶段的资源下载。另外我们做虚拟滚动，本质上是在 Layout/Paint 阶段减少实际参与布局的 DOM 数量。"

**项目印证**: `apps/vue-app/vite.config.ts:39-45` — viteCompression 插件；`apps/react-app/src/features/asset/components/VirtualTable.tsx` — 虚拟滚动。

### 2. 浏览器渲染原理：重排（Reflow）和重绘（Repaint）

**口语回答**:
"重排就是 Layout 阶段被重新触发，比如修改了元素的宽高、位置、display 属性，浏览器需要重新计算几何信息。重绘是 Paint 阶段重来，比如只改了 color、background，布局信息没变。重排一定触发重绘，重绘不一定重排。

在低代码设计器里，拖拽组件时频繁触发 addComponent、排序，如果每次操作都直接操作 DOM 就会大量重排。我们的做法是用 Vue 的响应式数据驱动视图——拖拽只是修改 schema.value 里的 components 数组，Vue 内部 diff 后做最小化 DOM 更新。这本质上就是批量更新预防频繁重排。

另外 FormConsumer（`apps/vue-app/src/views/lowcode/FormConsumer.vue`）中有 loading/submitting 状态控制，这些状态变化只触发局部重绘，不会引起整体重排。"

### 3. 浏览器缓存策略（强缓存 vs 协商缓存）

**口语回答**:
"强缓存由 Cache-Control 和 Expires 控制，命中后不发请求直接拿缓存，状态码 200 (from disk cache)。协商缓存用 Last-Modified/If-Modified-Since 或 ETag/If-None-Match，会发请求但服务器返回 304 表示没变。

我们项目中的静态资源文件名带了 hash——在 vite.config.ts 的 build.rollupOptions.output 里配置了 chunkFileNames 带 hash，JS 文件叫 'static/js/[name]-[hash].js'。文件内容不变则 hash 不变，浏览器可以直接用强缓存。这比单纯的协商缓存效率高得多。"

**项目印证**: `apps/vue-app/vite.config.ts:78-80` — `chunkFileNames/entryFileNames/assetFileNames` 配置。

### 4. 事件循环（Event Loop）

**口语回答**:
"JS 是单线程的。宏任务包括整体 script、setTimeout、setInterval、I/O；微任务包括 Promise.then、MutationObserver、queueMicrotask。每轮事件循环先执行一个宏任务，然后清空所有微任务，再检查是否需要渲染。

在表单消费端 FormConsumer.vue 的提交逻辑里，我们用了 async/await——`await formRef.value.validate()` 本质是 Promise，属于微任务队列。如果我要在验证完成后立刻更新 UI 状态（比如 submitting 从 loading 切换），放在 Promise.then 里可以保证在下一轮渲染前完成，比 setTimeout 更及时。"

**项目印证**: `apps/vue-app/src/views/lowcode/FormConsumer.vue:47-60` — handleSubmit 中的 async/await 链。

### 5. 跨域及解决方案

**口语回答**:
"跨域是浏览器的同源策略限制，协议、域名、端口任一不同就算跨域。常用方案有 CORS（服务端设 Access-Control-Allow-Origin）、JSONP、代理。

我们项目开发环境用 Vite 的 proxy 解决——在 `vite.config.ts` 里配置了 `/api` 路径转发到 `localhost:3000`，并且 `changeOrigin: true` 修改了请求头的 origin 字段，这样后端就感知不到跨域。生产环境通常用 Nginx 反代。"

**项目印证**: `apps/vue-app/vite.config.ts:87-93` — server.proxy 配置。

## 二、计算机网络八股

### 6. HTTP 1.1 / 2 / 3 区别

**口语回答**:
"1.1 的问题是队头阻塞——同一连接上请求必须排队响应；2.0 引入多路复用，一个 TCP 连接上可以并发多个请求和响应，同时有头部压缩（HPACK）和服务端推送。3.0 把传输层从 TCP 换成 QUIC（UDP），彻底解决 TCP 队头阻塞，0-RTT 握手更快。

我们的请求层封装在 `packages/shared/src/api/request.ts`，用的是 Axios。Axios 默认走 HTTP/1.1，但如果在 HTTP/2 环境下部署，它也同样受益于多路复用——表单消费端可能并发获取配置、提交数据、拉取角色权限，三个请求不需要三个连接。"

**项目印证**: `packages/shared/src/api/request.ts` — Axios 实例封装。

---

### 7. HTTPS 握手过程

**口语回答**:
"TLS 1.2 需要 2-RTT：Client Hello → Server Hello + 证书 → Client Key Exchange → 完成。TLS 1.3 简化为 1-RTT。核心是用非对称加密（RSA/ECDHE）交换对称密钥，后续数据用对称加密（AES）传输。

这个跟项目直接关系不大，但面试官如果追问安全，我会提到：我们的请求拦截器里从 localStorage 取 JWT token，挂到 Authorization 头——JWT 本身只保证内容的完整性和真实性，但安全性依赖 HTTPS 保证传输过程不被窃听。"

---

### 8. TCP 三次握手和四次挥手

**口语回答**:
"三次握手：客户端 SYN → 服务端 SYN+ACK → 客户端 ACK。为什么三次不是两次？因为要防止已失效的连接请求到达服务端，三次才能确认双方收发能力正常。

四次挥手：客户端 FIN → 服务端 ACK → 服务端 FIN → 客户端 ACK，多一次是因为 TCP 是全双工的，一方关闭后另一方可能还有数据要发。

实际开发中我更多关注长连接和连接池——Axios 的 keep-alive 默认开启，避免频繁握手的开销。"

---

### 9. CDN 原理与静态资源优化

**口语回答**:
"CDN 核心就两点：智能 DNS 把用户请求导向最近的边缘节点，内容缓存减少回源。

我们的构建产物中，echarts、element-plus 这些第三方库被拆成独立 chunk（echarts-vendor、element-plus-vendor），部署时可以把这些 chunk 推到 CDN 上。因为这些库版本变更频率低，CDN 缓存命中率高，用户二次访问几乎不需要重新下载。"

**项目印证**: `apps/vue-app/vite.config.ts:67-76` — manualChunks 配置。

## 三、JavaScript 基础

### 10. 闭包及应用场景

**口语回答**:
"闭包就是函数能记住并访问它被创建时的词法作用域，即使这个函数在别处被调用。

项目里最典型的应用场景是 `findComponentAndParent` 函数——它在低代码设计器中递归遍历 components 树查找某个组件节点（`apps/vue-app/src/views/lowcode/FormBuilder.vue:45`）。每次递归调用都在创建新的闭包，内层函数能访问外层的 list、i 变量。另外 Axios 的请求/响应拦截器也是闭包——拦截器函数记住了创建时的 config 和 error 上下文。"

---

### 11. 原型链和继承

**口语回答**:
"每个 JS 对象都有一个 `__proto__` 指向它的构造函数的 prototype，这样形成一条链。访问属性时沿着这条链往上查找。

说实话在工作中直接操作原型链的场景不多。TS 里更常用 class 语法糖。不过如果面试官追问，我会举这个例子：我们的 `WorkflowEngine` 类在 `packages/core/src/asset/workflow.ts` 中用 static 方法避免了实例化，这是一种设计模式的选择而非原型链的问题。真正的原型链应用场景是 Vue 3 的响应式内部实现——reactive 用 Proxy 拦截 get，get 中 track 依赖收集，这取代了 Vue 2 基于 Object.defineProperty 的原型拦截方式。"

---

### 12. Promise 和 async/await

**口语回答**:
"Promise 有三种状态 pending/fulfilled/rejected，状态不可逆。async/await 是 Promise 的语法糖，让异步代码读起来像同步。

FormConsumer 的 onMounted 里，我用 await 等待 API 返回表单配置，然后初始化 formData 和 formRules。这比 `.then()` 链式写法清晰太多。但要注意 await 会阻塞当前 async 函数的后续代码，如果两个请求没有依赖关系，应该用 `Promise.all` 并发。

另外提交逻辑中 `finally` 块重置 `submitting.value = false` 是关键——不管成功失败都要恢复 UI 状态，这比在 try 和 catch 里各写一遍好得多。"

**项目印证**: `apps/vue-app/src/views/lowcode/FormConsumer.vue:28-42` — onMounted 中的异步加载。

---

### 13. this 指向问题

**口语回答**:
"this 的指向取决于调用方式：普通函数指向调用者，箭头函数继承外层作用域的 this，call/apply/bind 可以显式绑定。

Vue 3 中用 `<script setup>` 语法糖时，其实不需要关心 this——setup 中没有 this。但如果在 Options API 中，methods 里的 this 指向组件实例。我们项目全用 Composition API + setup，所以 this 的问题被规避了。

唯一要注意的是回调函数——比如 FormConsumer 的 validate 回调中如果用普通函数，this 可能指向 el-form 实例而非组件。解决方案就是统一用箭头函数。"

---

### 14. 深拷贝与浅拷贝

**口语回答**:
"浅拷贝只复制一层引用，深拷贝递归复制所有层级。JSON.parse(JSON.stringify()) 简单但有坑——不能处理 undefined、函数、循环引用。

低代码设计器中有一个精妙的设计：`arrToTree` 函数在 `packages/core/src/asset/tree.ts` 里用 `{ ...i, children: [] }` 做浅拷贝，因为源数据是平铺列表，树结构是派生数据，浅拷贝足够且避免了不必要的深拷贝开销。

如果面试官问大数据场景怎么办，我会说：在资产管理模块的 VirtualTable 中（`apps/react-app/src/features/asset/components/VirtualTable.tsx`），我们不拷贝数据，而是计算可视窗口索引，只渲染那部分行——连浅拷贝都省了。"

---

### 15. 防抖与节流

**口语回答**:
"防抖是延迟执行，连续触发只执行最后一次（搜索框输入）。节流是固定频率执行（滚动事件、resize）。

低代码设计器右侧属性面板里，用户修改 input 的 placeholder、label 等配置时，每个字段变化都触发 schema 更新。虽然没有显式写 debounce，但 Vue 的异步更新队列本质上就是一种批量防抖机制——多次数据变化合并成一次 DOM 更新。如果面试官追问为什么不加 debounce，我会说这是设计选择：属性面板的实时预览需要即时反馈，延迟反而显得卡顿。"

## 四、React/Vue 框架结合项目

### 16. Vue 3 响应式原理

**口语回答**:
"Vue 3 用 Proxy 替代了 Vue 2 的 Object.defineProperty。Proxy 可以直接拦截对象属性的 get/set/deleteProperty 等 13 种操作，不需要像 Vue 2 一样递归遍历属性的 getter/setter。

在低代码设计器中，schema 是一个 `ref<FormSchema>`。当用户拖拽添加组件时，我们往 `schema.value.components` push 一个节点对象。Proxy 拦截这个 push 操作，触发依赖更新，画布自动重新渲染。这比 Vue 2 需要 `Vue.set` 才能检测数组变化要好得多。

另外 Pinia 的响应式也是基于 Vue 3 的 reactivity 包。我们的 formBuilder 没有用 Pinia 管理 schema，而是用组件内 ref，因为 schema 只在 FormBuilder 和 FormConsumer 之间流转，不需要全局共享。这是一种权衡——不是所有状态都值得放进 store。"

**项目印证**: `apps/vue-app/src/views/lowcode/FormBuilder.vue:17-22` — schema ref 定义。

---

### 17. Vue 3 Composition API vs Options API

**口语回答**:
"Composition API 的优势是逻辑复用——可以把相关代码（数据、计算属性、方法、watch）放一起，而 Options API 按选项类型分散在不同位置。

项目从根到叶子全部用的 `<script setup>`。比如 FormBuilder.vue，拖拽逻辑、删除逻辑、代码生成逻辑各自集中，不会散落在 data/methods/computed 各处。而且配合 TypeScript 的类型推导，ref 和 reactive 的类型都能自动推断，不需要像 Options API 那样手动声明返回类型。"

---

### 18. React Hooks 与 Vue Composition API 对比

**口语回答**:
"这是面试高频题。相同点：都解决了逻辑复用问题。不同点：

1. 响应式机制：React 依赖显式的依赖数组（useEffect、useMemo、useCallback），漏了依赖就出 bug；Vue 自动追踪，不需要手动声明。
2. 渲染机制：React 组件每次状态变化都重新执行函数体，Hooks 必须保持调用顺序不变；Vue 的 setup 只执行一次，响应式通过 ref/reactive 的 getter/setter 工作。
3. 心智负担：Vue 更符合直觉，React 需要理解闭包陷阱和 stale closure。

我在同一个 Monorepo 里同时维护 Vue 3 和 React 19 的低代码引擎，这让我很清楚地感受到两者的差异。React 的并发特性（useTransition、Suspense）在处理大数据渲染时优势明显，但简单的表单场景 Vue 写起来更快。"

**项目印证**: `apps/vue-app` 和 `apps/react-app` 的双框架架构；`apps/react-app/src/features/asset/components/VirtualTable.tsx` — React 虚拟滚动。

### 19. Vue Router 动态路由和权限控制

**口语回答**:
"我们用了静态路由 + 动态路由的分层策略。constantRoutes 包含 login 和 404，任何人可访问。asyncRoutes 在 meta 里定义了 roles 字段，登录后根据用户角色过滤。

关键实现：路由守卫在 beforeEach 里判断 token 是否存在，没 token 就跳转 login。登录成功后调用 `router.addRoute` 将过滤后的 asyncRoutes 动态添加。侧边栏菜单也是根据 roles 动态生成——permission store 里的 generateRoutes 会递归过滤权限树。

还有一个容易忽略的点：KeepAlive。低代码设计器页面加了 `KeepAlive: true`，用户切到其他页面再回来，canvas 上的组件不会重置。这对表单设计的用户体验很重要。"

**项目印证**: `apps/vue-app/src/router/index.ts:11-80` — constantRoutes/asyncRoutes 定义，KeepAlive 配置。

### 20. Pinia 状态管理设计

**口语回答**:
"Pinia 相比 Vuex 简化了很多——没有 mutations，只有 state/getters/actions，TS 支持也更好。

我们没有把所有状态都塞进 Pinia。低代码设计器的 schema 用组件内 ref 是因为它只在 FormBuilder 内部流转。Pinia 主要管理跨组件/跨页面的状态——比如用户信息、权限列表、侧边栏折叠状态。

我会强调一个设计原则：状态提升要看通信距离。父子通信用 props/emit，兄弟通信找共同父组件，跨页面共享才进 store。这样可以避免 store 变成垃圾场。"

### 21. React 虚拟列表（VirtualTable）

**口语回答**:
"这是简历上 SaaS 资产管理系统的核心亮点。传统表格用 `Array.map` 渲染全部数据行，当数据超过 5000 条时浏览器直接卡死。虚拟列表的思路是：只渲染可视区域内的行。

具体做法：用一个撑开高度的幽灵容器（Phantom Height）来模拟完整内容的高度，让滚动条正常工作。监听 scroll 事件计算当前可视区的起始索引和结束索引，只渲染这个窗口内的行，并通过 `translateY` 偏移定位。

难点在于：React 19 的 concurrent rendering 可能中断虚拟列表的计算，导致闪烁。解决方案是使用 `useDeferredValue` 或者手动维护 startIndex/endIndex 的 ref，避免并发中断问题。"

**项目印证**: `apps/react-app/src/features/asset/components/VirtualTable.tsx`。

### 22. 低代码引擎的 AST 代码生成

**口语回答**:
"这是项目中最有技术含量的模块。传统做法是字符串拼接模板——'<' + type + ' v-model="' + field + '">'。这种方式的问题是：容易拼接错误、无法做语法校验、不支持复杂的嵌套结构。

我们的 SimpleASTGenerator 在 `apps/vue-app/src/utils/simpleAstGenerator.ts` 中实现。它不是真正的编译器，但借鉴了 AST 的思想：

1. 遍历 components 树（自顶向下递归）
2. 根据组件类型分派到不同生成器（generateFormItem / generateGroupItem / generateGridItem）
3. 处理数据扁平化——嵌套组件中的 v-model 需要正确计算绑定路径，避免 depth > 1 时的路径错误
4. 生成完整的三段式 Vue SFC（template + script setup + scoped style）

我们正在从这个简化版升级到真正的 AST 解析器，目标是用 @babel/parser 生成标准 AST 再做代码转换，这样可以借助 Babel 生态做代码优化和静态分析。"

**项目印证**: `apps/vue-app/src/utils/simpleAstGenerator.ts`；`AST_GENERATOR_UPGRADE.md`。

## 五、性能优化

### 23. 首屏加载优化

**口语回答**:
"首屏优化我一般从四个维度入手：

**1. 资源体积**：vite-plugin-compression 生成 Gzip，Element Plus 通过 unplugin-vue-components 按需引入而非全量导入。构建分析用 rollup-plugin-visualizer 看哪个包最大。

**2. 代码分割**：manualChunks 把 echarts、element-plus 拆成独立 chunk，vendor 放其他第三方库。路由懒加载用 `() => import('../views/xxx.vue')`，配合 KeepAlive 缓存已加载页面。

**3. 网络优化**：开发环境 proxy 代理，生产环境 CDN + HTTP/2。

**4. 渲染优化**：虚拟列表处理长列表，首屏只加载可见内容。

面试官如果问具体数据，我会说：我们项目的构建产物从初始 3.5MB 压缩到 Gzip 后约 420KB，通过 chunk 拆分后首屏只需加载 vendor + 首页 chunk，约 180KB。"

**项目印证**: `apps/vue-app/vite.config.ts` — 完整构建优化配置。

### 24. 长列表渲染优化

**口语回答**:
"虚拟列表是核心方案。我在 React 端的 SaaS 系统中手写了 VirtualTable 组件：
- 固定行高方案：itemHeight * totalCount 算总高度，scrollTop / itemHeight 算起始索引
- 动态行高方案：维护一个行高缓存表，用预估高度撑开滚动条，实测后再修正
- 用 IntersectionObserver 做懒加载替代 scroll 事件监听，减少主线程占用

另一个优化点：Vue 端的低代码设计器中，components 数组可能很深（嵌套分组+栅格）。渲染用的 `v-for` 配合 `:key` 确保 diff 高效。如果面试官追问为什么不用 v-if 替代，我会说 v-if 有编译时优化（PatchFlags），但列表渲染场景下 v-for + key 是正确的选择。"

### 25. 图片优化

**口语回答**:
"图片优化主要三点：格式选 WebP（比 PNG 小 30%），响应式图片按屏幕密度加载不同尺寸，懒加载减少首屏请求。

坦白说，我们的低代码引擎主要是表单和表格，图片不是重点。但在资产管理系统的设备图片展示中，可以用 Vite 的 `import` 动态导入 + 构建时压缩。如果面试官追问大图场景，我会提 CDN + 渐进式加载的方案。"

### 26. 内存泄漏排查

**口语回答**:
"前端内存泄漏常见原因：未清除的定时器/事件监听、闭包引用了大对象、销毁的组件未取消订阅。

排查手段：Chrome DevTools 的 Memory 面板，先拍 heap snapshot 基线，操作后再拍一个，对比两次的 Delta 看哪些对象没被回收。

我们的低代码设计器中有一个潜在风险：递归组件 `BuilderNode` 和 `FormNode` 是自引用的——`list` 里可能再嵌套自己。如果组件卸载时没有正确清理，递归引用可能导致 GC 无法回收。解决方案是在 `onBeforeUnmount` 中把 list 置空，断开引用链。"

**项目印证**: `apps/vue-app/src/components/BuilderNode.vue`、`FormNode.vue` — 递归组件。

## 六、代码排查与调试

### 28. 线上 Bug 排查流程

**口语回答**:
"我的排查流程：复现 → 定位范围 → 二分法缩小 → 读源码 → 修复。

1. 先确认环境：哪个浏览器、哪个版本、什么操作路径。前端 Bug 很多是环境差异导致的。
2. 利用 Chrome DevTools：Network 看接口返回是否正确，Console 看报错堆栈，Sources 打断点追踪数据流。
3. 如果是接口问题：排查 Axios 拦截器有没有吞错误——我们项目里 401 拦截器会跳转 login 并 reject，但 Error 对象被消费后不再上报，可能导致静默失败。
4. 如果是渲染问题：用 Vue DevTools 看组件树，检查 props 传递是否中断、响应式是否丢失。

真实踩坑：低代码设计器中 select 组件的 options 更新后下拉列表没刷新。排查发现是 options 用了 ref 的深层赋值但没有触发响应——因为 options 是数组，replace 整个数组引用才触发更新，push 等变异方法在深层对象上需要额外处理。"

### 29. console.log 之外的高级调试手段

**口语回答**:
"1. **断点调试**：Sources 面板设条件断点（右键 → Add conditional breakpoint），只在特定条件下暂停，比如组件 id 匹配时。

2. **Performance 面板**：录制操作过程，看 FPS、Long Task。虚拟列表的滚动卡顿就是用这个定位的——发现每次 scroll 都有 50ms 的 recalculate style。

3. **Vue DevTools**：直接看组件的 props、data、computed 值，还能手动触发事件。排查响应式问题时非常有用。

4. **Network 节流**：模拟弱网环境（Slow 3G），测试 loading 状态和错误处理。我们 FormConsumer 的 loading/skeleton 就是在这种场景下验证的。

5. **覆盖率和源码映射**：配合 source map，线上压缩代码也能定位到原始文件行号。Vite 默认 dev 模式开 source map。"

### 30. 排查接口 401/403 问题

**口语回答**:
"从请求链路两端排查：

**前端侧**：检查 request interceptor 是否正确添加 token。我们的 `packages/shared/src/api/request.ts` 里从 localStorage 读 'auth-storage'。如果 localStorage key 拼错了或者 JSON 结构变了（比如用了 zustand 的 persist 中间件导致结构不同），token 就不会被带上。

**后端侧**：确认 JWT 是否过期，过期时间配置是否合理。我们的处理是 401 时清除 auth-storage 并跳转 login。

**中间层**：如果用了 Nginx 代理，检查是否把 Authorization 头过滤掉了（某些 Nginx 配置默认不带下划线头）。

一个高频坑：同域名多 Tab 登录不同账号。一个 Tab 的 401 清除 localStorage，导致其他 Tab 也被踢出。解决方案是在 token 前面拼用户名做隔离，或者用 sessionStorage。"

### 31. TypeScript 类型报错排查

**口语回答**:
"TS 报错本质上是类型不匹配。我的排查习惯：

1. 从报错行往上找，看是哪里的类型定义和实际值不一致。
2. 用 VSCode 的 hover 查看变量推导类型，往往发现问题在于「你以为的类型」和「TS 推导的类型」不一样。
3. 检查泛型传递链是否中断——比如 `filterAssets<T extends AssetProject>`，如果传入的数组元素类型是 any，整个链就断了。

项目案例：FormSchema 的 components 类型原本是 `FormComponent[]`，但递归组件中 `list` 字段也是 `FormComponent[]`，这导致 TS 在递归类型推断时可能进入无限展开。解决方案是用 interface 而非 type 定义，TS 对 interface 的递归有更好的处理。"

**项目印证**: `apps/vue-app/src/types/lowcode.ts` — TypeScript 类型定义。

### 32. 递归组件调试

**口语回答**:
"低代码引擎中 BuilderNode 和 FormNode 都是递归自引用组件——组件内部又调用自己来渲染子节点。

调试难点在于调用栈很深，一个断点可能被重复命中上百次。我的技巧：

1. 在组件的 props 解析处设条件断点，比如 `comp.id === 'target-id'`，只在排查的目标节点暂停
2. 用 console.group/groupEnd 在递归函数入口和出口打日志，形成可折叠的层级输出
3. 限制递归深度——写一个 maxDepth 参数，超过就抛错，防止死循环

实际踩过坑：group 套 group 时 `findComponentAndParent` 递归查找失败的节点被静默忽略，导致拖入第二个 group 时不显示。调试方法是打印每一步的 list 和 index，发现是 pid 引用断裂——问题不在递归算法，在数据源的 id 生成逻辑。"

**项目印证**: `apps/vue-app/src/views/lowcode/FormBuilder.vue:45-53` — findComponentAndParent 递归查找。
