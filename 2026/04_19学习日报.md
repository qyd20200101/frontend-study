    + 作用域与闭包
        + 作用域链：执行代码，查找变量的路径
        + 闭包：函数在其外部作用域执行，保存其定义的作用域和变量

        + 变量提升：var,function声明会提升，let/const存在暂时性死区
    + 浏览器渲染流程
        + 渲染路径：html到DOM,css——CSSOM——Render tree——Layout(回流)——paint(重绘)
        + 回流：元素尺寸，布局发生变化,width,height,fontSize宽高字体等发生改变
        + 重绘：元素外观如颜色背景颜色发生改变，但是布局不变的
        + 虚拟列表渲染5万条数据不会卡死：
            + 直接渲染：浏览器计算5万DOM的layout，线程被撑爆
            + 虚拟：transform是在合成层处理，不触发回流和重绘，直接又GPU加速
    + 计算机网络

            + 三次握手：客户端——>服务端——>客户端,关键标志位：“SYN（请求连接）-> SYN+ACK（确认并响应）-> ACK（最终确认）”。
            + 四次挥手：断开链接，因为服务端可能有数据没传完
            + TCP重传机制，拥塞控制，UDP快（直播，游戏用）

+ 心得与收获： 总结学习成果，要详细说明对相关知识点的个人理解。
    + 对 Flex 布局陷阱的深刻认知：今天最核心的收获是明白了为什么设置了 height: 100% 的元素会变成“窄纸条”悬浮在半空中。这是因为一旦父容器开启了 align-items: center（交叉轴居中），子元素原本继承的高度就会被强制压缩到内容本身的真实高度，导致 100% 完全失效。这在手写复杂表格时是一个极其隐蔽的雷区。

    + 组件解耦的“纯净度”至关重要：以前总习惯在封装公共组件（如 VirtualTable）时顺手写上 padding、border 和 flex 居中。今天才痛彻心扉地领悟到：真正的底层组件不应该干涉排版。底层只管生成确定高度的 div 坑位，对齐、边框、背景色必须完全交给外层业务组件（通过插槽传入）去控制。解除了“双重武装”后，样式冲突迎刃而解。

    + CSS 样式污染的排查思路：遇到样式问题，不能盲目地使用 !important 或内联样式去“暴力镇压”。最优雅的做法是审查 DOM 树，找到产生污染的源头属性（如幽灵 line-height 或重复的 display: flex），从根源上拔除病灶。
+ 遇到的问题 (可选)： 有什么问题
    + 问题描述：列表行内组件和边框出现诡异的“悬浮（浮动）”错位现象，操作按钮上飘，整行背景色没有撑满 60px 的行高，像小纸条一样飘在中间。

    + 产生原因：这是典型的“双重布局冲突”。底层 VirtualTable 的包装盒设置了 flex 居中，而外层 DataManager 传进去的 .table-row 也设置了 flex 居中。父级的居中属性把子级 .table-row 的物理高度给挤压塌陷了。

    + 解决方案：对 VirtualTable.vue 进行“大清洗”，删除了原有的 display: flex、align-items: center、line-height 和 padding，只保留高度占位。将所有居中对齐和 100% 高度填充的控制权完全交还给 DataManager.vue 的 .table-row，最终实现了像素级的完美垂直居中和底边框对齐。
