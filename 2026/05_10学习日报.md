## 2026/05/10 Day11 Vue3深度学习日报

+ 学习时长：3 小时

+ 学习内容：
  - 递归组件事件冒泡补完：区分原生 DOM 冒泡（.stop 阻止）和 Vue emit 链（每层显式监听 @select 并转发），两者互不干扰
  - v-model 语法糖完整链路：v-model="x" = :modelValue="x" + @update:modelValue="x=\"，结合 FormNode.vue 推导用户输入→el-input emit→formData[comp.field]赋值→响应式系统 trigger→scheduler 批量调度→patch 更新 DOM 的完整 7 步链路
  - Pinia 本质：defineStore 是 setup 函数的全局单例，任何组件 useUserStore() 拿到同一实例，类比 Day 1 手写的 effect/reactive 共享同一套 track/trigger
  - KeepAlive 闭环：:include="cacheViews" 动态控制缓存名单，addView 加入缓存，delView 移除缓存，切走触发 onDeactivated（不销毁），切回触发 onActivated（不重新挂载）
  - 编译原理三件套：静态提升（不变 VNode 提到 render 外部）、PatchFlag（标记动态节点，diff 跳过静态）、事件缓存（_cache 复用闭包），v-if→三元表达式，v-for→.map()
  - 响应式源码链路：effect→track(收集)→trigger(派发)→scheduler(去重批量)→effect.run()(render→patch→DOM)
  - 挂载渲染链路：createVNode→mount→patch(null)→setupComponent→setupRenderEffect(effect)
  - 12 题自检测验，得分 8/12

+ 心得与收获：
  今天最大的收获是用自检暴露了三个盲区：① 静态提升和 PatchFlag 的职责边界——静态提升管"创建"（VNode 只创建一次），PatchFlag 管"比对"（diff 时跳过未标记的静态属性），之前一直把两者混为一谈；② TS 工具类型的写法——Partial<SystemUser> 是内置工具类型直接传泛型参数，不是 type<Partial extends SystemUser> 这种不存在的语法；③ v-if/v-else-if/v-else 编译成三元表达式的确切形态（a ? b : c ? d : e）。
  KeepAlive 和 Pinia 的结合使用让我理解了"数据在哪，缓存到哪"的设计思路——cacheViews 不是静态配置，而是运行时通过 addView/delView 动态维护的名单，用户打开和关闭 Tab 的行为实时影响 KeepAlive 的缓存范围。
  编译原理的学习让之前的很多疑惑解开了——为什么 Vue 3 比 Vue 2 快？不是某个单一优化，而是静态提升+PatchFlag+事件缓存三件套叠加，分别解决了"重复创建""全量比对""重复创建闭包"三个独立问题。

+ 错误点与加强方向：
  - 【编译层】静态提升 vs PatchFlag 容易混淆 → 加强：静态提升管 VNode 创建，PatchFlag 管 VNode 比对，两者是编译阶段和 diff 阶段的分工
  - 【编译层】v-if/v-else 编译结果 → 加强：记住"三元表达式"这个关键词，理解为什么不用 if-else（render 函数需要返回值）
  - 【TS 层】Partial 语法写法错误 → 加强：工具类型本质是类型函数，语法是 Partial<Type> 而非 extends 语法
  - 【TS 层】extends 在泛型中的作用 → 加强：extends 是泛型约束（P 必须满足 BasePageParams 的结构），不是类型收缩
  - 【响应式层】需要继续巩固 track/trigger/scheduler 的完整调用时序，尤其 scheduler 的去重机制（同一 effect 不重复入队）
