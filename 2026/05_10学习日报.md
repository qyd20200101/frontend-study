## 2026/05/10 Day12 Vue3 深度学习日报

+ 学习时长：3.5 小时

+ 学习内容：
  - provide/inject 跨层级通信：用 Symbol + InjectionKey 封装 useBuilderActions hook，改造 BuilderNode.vue 将 select/delete/dragstart/drop 四个事件从逐层 emit 转发改为 inject 直达，删除 defineEmits 和中间层转发代码，FormBuilder.vue 通过 provide(BUILDER_KEY, {...}) 注入方法
  - scheduler 源码原理：trigger 不直接执行 effect，而是通过 queueJob 入队——Set 自动去重 + Promise.resolve() 微任务批量执行，同一 effect 不重复入队，三次赋值只跑一次 render
  - nextTick 与 scheduler 的关系：两者共用同一个微任务队列，scheduler 的 effect 排在前面执行，nextTick 回调排在后面，因此 nextTick 能拿到更新后的 DOM
  - diff 算法核心：双端对比快速锁定头尾相同节点缩圈，中间乱序部分通过最长递增子序列找出不需要移动的节点，只移动其余节点；重新强调静态提升/PatchFlag 是编译阶段优化，diff 是运行时消费者
  - key 的作用：唯一标识节点，diff 时判断复用还是重建，key=index 在排序时会导致全量 DOM 重建和状态丢失
  - v-permission 自定义指令：mounted 钩子在元素插入 DOM 后触发，updated 钩子响应运行时权限变化，指令不响应式的局限性及组件替代方案
  - 指令 7 个生命周期钩子：created→beforeMount→mounted→beforeUpdate→updated→beforeUnmount→unmounted，指令 mounted 晚于组件 onMounted 触发
  - 插件系统 app.use() 原理：调用插件的 install(app) 方法注册全局能力；pinia.use() 操作的是 Pinia 实例而非 app 实例
  - TS 修补：Pick<T, K> 挑字段、Partial<T> 全可选、extends 在泛型中作约束而非继承

+ 心得与收获：
  provide/inject 改造是今天最大的实战收获。之前的 BuilderNode.vue 中每层都要写 4 个 @select/@delete/@dragstart/@drop 转发事件，3 层嵌套就要写 12 行重复代码。改造后用 Symbol + InjectionKey 封装成 useBuilderActions hook，中间层不再关心事件，只有 Root 通过 provide 注入一次，任何深度的 BuilderNode 直接 inject 调用。解构写法使得模板中 onSelect(comp.id) 比 actions.onSelect(comp.id) 更简洁。
  scheduler 的学习让我把 Day 1 手写的 mini ref 和真实源码串起来了——之前 trigger 直接 dep.forEach(fn => fn())，真实 Vue 里加了 queueJob 中间层做去重和批量调度，Set 比数组更适合做去重队列。nextTick 不再是神秘的黑盒，它就是排在 scheduler effect 后面的微任务回调。
  diff 算法的双端对比 + 最长递增子序列两步走模式让我理解了为什么 Vue 3 拖拽排序性能好——先快速锁定不变的头部尾部，中间只处理真正乱的部分。但今天再次暴露了"静态提升/PatchFlag 归入 diff"的惯性错误，需要时刻区分"编译时优化"和"运行时比对"两个阶段。
  自定义指令的局限性也清楚了——指令只在 mounted 和 updated 一瞬间执行，不具备响应式；适合一次性判断（如权限），不适合需要持续响应数据变化的场景（此时应改用组件包裹）。

+ 遇到问题和加强方向：
  - 【顽固盲区】静态提升/PatchFlag 第三次被误归入 diff 算法，需牢记：编译阶段产出标记，diff 阶段消费标记。口诀：编译管创建，diff 管比对
  - 【薄弱点】指令 mounted 和组件 onMounted 的先后顺序需巩固：组件 onMounted 先，指令 mounted 后
  - 【TS 层】工具类型写法已从 type<Partial extends T> 纠正为 Partial<T>，但需要更多实战练习强化
