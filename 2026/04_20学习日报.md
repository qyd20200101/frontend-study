        + 协商缓存，响应头：ETag,Last-Modified浏览器通过ETag询问服务器资源是否改变，返回304则没改变继续使用旧的，节约带宽和解析时间
        + ETag:主要检查内容发生改变,Lat-Modified检测时间，如果一秒钟多次修改感知不到

        + 地址栏回车：强缓存
        + F5刷新，失效强缓存，走协商缓存，跳过Cache-Control
        + Ctrl+F5，所有缓存失效，从服务器下载资源
    + Vite打包，通过index[hash].js检测哈希值是否改变，改变就下新文件
        + HTML文件设no-cache每次都协商，JS/CSS设置max-age=1年,强缓存

+  心得与收获： 总结学习成果，要详细说明对相关知识点的个人理解。
    + 写弹窗只无脑用 position: fixed 铺满全屏。今天实战后才真正理解，absolute 定位其实非常听话，只要给它的父容器上一把 relative 的“锁”，它就会乖乖呆在指定的局部空间里。这是提升 B 端系统多线操作体验的神技。

    + 网络请求“防抖”的降维打击：之前都是在 UI 按钮上写防抖，但今天在 Axios 层实现 AbortController 拦截后，感觉像给系统穿上了防弹衣。看着浏览器的 Network 面板里前置的重复请求直接标红变成 (canceled)，对前端基建的掌控感直线上升。

    + HTTP 协议的传参规矩：在排查批量删除和分页 Bug 时，深刻体会到了 RESTful 规范的严谨性。GET 请求没有 Body，参数必须老老实实挂在 URL 后面让后端从 req.query 拿；而 DELETE/POST 操作才把数组放进 req.body 里。这种前后端联调的“暗坑”，踩过一次就永远记住了。

    + KeepAlive 的生命周期陷阱：引入缓存后，页面不再频繁销毁，导致挂载在 window 上的 resize 事件疯狂触发引发内存泄漏。及时将事件监听从 onMounted/onUnmounted 迁移到专属的 onActivated/onDeactivated 中，让我对 Vue 的生命周期有了更立体的认识
+  遇到的问题 (可选)： 有什么问题
  问题1：配置了 <KeepAlive> 后，页签切换依然无法保留表单输入内容。

  +  解决：排查发现 <keep-alive> 的 include 属性极度严格。通过在 .vue 文件顶部增加传统的 <script lang="ts"> 显式声明 name，并确保与 router/index.ts 中的名字大小写完全一致，成功激活了缓存魔法。

+ 问题2：列表多选框选中后点击“批量删除”，参数传不到后端，且分页查询报错 500。
  + 解决：首先在 TS 层面修复了接口定义的入参类型约束（接收 number[]）。其次在 Express 后端修正了取参方式：GET 分页从 req.query 结构参数并处理 NaN 容错，DELETE 批量操作从 req.body 接收数组。前后端数据流终于完美闭环。
+ 问题3：TypeScript 报错提示 { page: number, pageSize: number } 不能赋给泛型 P。
  + 解决：这是 TS 保护机制，无法判定外来泛型 P 是否还需要其他必填属性。采取了官方推荐的双重断言 as unknown as P 强制兼容，既保留了泛型推导，又排除了编译器报错。
