  + 编写并全局注册了自定义指令 v-permission，实现了基于角色（Roles）的 DOM 元素物理移除。
+ 深入理解了声明式权限控制中“组件式（AuthButton）”与“指令式（v-permission）”的应用场景区别。
+ Mock Server 架构加固与 Bug 调试：
  + 定位并修复了因 unshift 操作导致 admin 账户索引位移，从而引发 /api/user/info 返回错误身份的逻辑漏洞。
  + 优化了 Mock 后端的身份识别逻辑，从“写死索引”进化为“基于请求头 Token 匹配用户”。
  + 改进了 ID 生成策略，将不稳定的 Date.now() 替换为更具规范性的随机哈希 ID 字符串。

+ 系统架构重构与功能迁移：
  + 遵循“关注点分离”原则，将用户管理功能从业务层（DataManager.vue）完整迁移至系统层（System.vue）。
  + 完善了用户管理全链路，包括新增用户的密码设置、后端密码校验以及前端表单规范化。

+ TypeScript 高级类型应用与报错排查：
  + 解决了 string | number 类型不重叠导致的 API 参数报错，同步更新了 api/user.ts 的类型契约。

+ 学习了通过可选属性（?）处理“新增实体”与“现有实体”之间字段差异的技巧。

+ 心得与收获：
  + 对前端权限的深度理解：
  + 前端权限（如 v-permission）本质上是 UI 层面的交互优化，通过 el.parentNode.removeChild(el) 实现物理移除比 display:none 更专业且不易被绕过。但真正的安全性必须在后端（如 server.js 接口校验）形成闭环。
  + 架构思维的提升：
  + 以前习惯把功能堆在一起，今天意识到 DataManager 应该纯粹处理资产业务，而 System 处理行政管理。职责清晰的代码在面对 TS 类型报错时，更容易定位是因为“接口定义过窄”还是“逻辑本身错误”。
  + 关于 Mock 数据的反思：
    即便是模拟后端，其逻辑也必须具备一定的“真实性”。今天因 unshift 导致的 Bug 让我明白，底层数据结构的改变（如数组顺序）会直接影响到基于索引的顶层逻辑，因此在处理“当前登录人”这种核心状态时，应优先使用唯一的标识符（如 Token 或 ID）进行查找。
  + TypeScript 的约束力：
    虽然 as unknown as SystemUser 可以快速解决报错，但最好的做法是在 interface 定义阶段就预见到业务的复杂性（如 ID 可能是多类型的），保持类型定义的灵活性和准确性。

+ 遇到的问题：
  类型冲突问题：在执行 deleteUserApi(row.id!) 时，因为 id 的类型变更为 string | number，导致与 API 原定义的 number 冲突。

+ 解决过程：通过同步修改 src/api/user.ts 中的参数定义，确保了全链路类型的一致性，同时也学习了 ! 非空断言在已确认数据存在场景下的正确用法。
